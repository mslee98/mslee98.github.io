# Monorepo vs Multirepo 장단점

## Monorepo

### ✅ Monorepo 장점

#### 1. 의존성 관리가 쉽다

모든 프로젝트가  
**하나의 의존성 그래프를 공유**한다.

- 동일 라이브러리 중복 설치 방지
- 버전 불일치로 인한 이슈 감소
- `Yarn Workspace` / `pnpm workspace` 캐시 공유

프론트 프로젝트를 여러번 운영해본 사람이라면  
의존성 버전 문제로 고생한 경험이 한 번쯤은 있을 것이다.

---

#### 2. 공통 컴포넌트(디자인 시스템) 관리가 편하다

멀티레포 환경에서도 공통 코드를 사용할 수는 있다.

하지만 대부분의 경우 아래 과정을 거친다.

- 공통 패키지 분리
- npm 배포
- 버전 관리
- 각 서비스에서 다시 설치

이 과정 자체가 문제는 아니지만,
`Storybook` 기반의 디자인 시스템을 운영하는 상황이라면 이야기가 조금 달라진다.

각 서비스에서 스타일이나 컴포넌트를 조금씩 개별 수정하기 시작하면,
어느 순간 디자인 시스템과 실제 서비스 UI가 어긋나기 시작하고
결국 “이게 최신이 맞나?” 하는 동기화 문제를 다시 겪게 된다.

반면 모노레포에서는  
공통 코드를 패키지로 분리한 뒤  
**ESM 방식으로 바로 import**할 수 있다.

```ts
import { Button } from '@shared/ui';
```
- 배포 필요 없음
- **수정 사항들이 즉시 반영!**
- UI / 유틸 / 타입 공유가 매우 수월

👉 아무래도 **Monorepo**가 많이 발전하면서 **디자인 시스템**도 같이 발전하는거 아닐까?

:::info 다만 개인적인 생각으로는,
디자인 시스템은 결국 라이브러리 형태로 가져가는 게 더 잘 맞는 방향이라고 생각한다.

토스의 **Deus**처럼 전용 툴을 사용하는 경우가 아니라면,
**Chromatic** + **Zeplin** + **Figma** + **Storybook** 조합만으로도
기획자, 디자이너, 퍼블리셔 간 협업이 꽤 자연스럽게 흘러간다.
실제로 UI 오차도 눈에 띄게 줄어드는 편이다.

여기에 더해
- 크로스 브라우저 체크
- 접근성 및 UI 테스트
- Git 브랜치 단위로 변경 사항을 시각적으로 확인하는 과정

같은 작업들도 비교적 수월하게 진행할 수 있다.

이런 흐름을 고려해보면,
디자인 시스템을 서비스 레포지토리에 무리해서 포함시키기보다는
아예 별도의 라이브러리로 분리해 배포하는 방식이 더 낫지 않을까라는 생각이 들었다.
:::

---

#### 3. 개발 생산성과 편의성

다음과 같은 설정들을  
각 프로젝트마다 반복할 필요가 없다.

- `eslint`
- `prettier`
- `TypeScript` 설정
- `tsconfig path`
- 공통 스크립트

👉 **Root 레벨에서 한 번에 관리 가능**

개발 환경을 맞추는 데 드는 비용이  
확실히 줄어든다.

---

### ⚠️ Monorepo 단점

#### 1. 초기 설정 비용이 크다

- Workspace 구조 설계
- 패키지 간 의존성 설계
- 빌드 / 테스트 범위 분리

아무 생각 없이 시작하면 중간중간 다시 손봐야 할 부분이 계속 생긴다.

> 모노레포는 "일단 만들어두고 쓰는 구조"가 아니다. 병렬적으로 CI를 처리한다든가 CI/CD 수준에서도 고민을 많이 해봐야한다.

---

#### 2. 저장소가 비대해진다

여러 프로젝트가  
하나의 Repository에 모이다 보니  
저장소 크기가 커질 수밖에 없다.

- clone 시간 증가
- IDE 인덱싱 부담
- 대규모 조직에서는 성능 이슈 발생 가능

## Multirepo

### ✅ Multirepo 장점

#### 1. 구조가 단순하다

- 하나의 서비스 = 하나의 Repository
- 이해하기 쉽고 진입 장벽이 낮다

소규모 팀이나  
단일 서비스에는 여전히 강력한 선택지다.

---

#### 2. 서비스 간 완전한 독립성

- 배포 주기 분리
- 의존성 완전 분리
- 장애 전파 최소화

한 서비스의 문제가  
다른 서비스에 영향을 주지 않는다.

---

### ⚠️ Multirepo 단점

#### 1. 공통 코드 관리가 어렵다

공통 코드 변경 시  
다음 과정을 반복하게 된다.

- 라이브러리 수정
- 버전 업데이트
- npm 배포
- 각 서비스에서 다시 설치

이 과정이 반복되다 보면  
결국 **복붙과 파편화**로 이어지기 쉽다.

---

#### 2. 의존성 버전 관리가 까다롭다

각 Repository가  
독립적으로 의존성을 관리하다 보니  
같은 라이브러리를  
서로 다른 버전으로 사용하는 일이 잦다.

## 마무리

Monorepo와 Multirepo는  
서로 우열을 가리는 관계가 아니다.

> **Monorepo는 관리 효율을 선택하는 것이고**  
> **Multirepo는 독립성을 선택하는 것이다.**

프로젝트 규모와 팀 구조를 고려해  
가장 잘 맞는 방식을 선택하는 것이 중요하다.

나처럼  
"플랫폼 팀은 없지만, 구조는 갖추고 싶다"면  
혼자서라도 모노레포를 경험해보는 건  
충분히 의미가 있다고 생각한다.
