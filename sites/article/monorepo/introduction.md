# 모노레포란 무엇인가?

**Monorepo**는 여러 개의 서비스 또는 패키지를 **하나의 Repository에서 관리하는 방식**이다.

반대로, 서비스마다 Repository를 분리해 관리하는 방식은 **MultiRepo**라고 부른다.

**Monorepo**가 왜 거론되는걸까?          
마이크로프론트엔드는 웹 애플리케이션을 독립적으로 개발하고 배포하며 운영 가능한 작은 단위로 분할하는 아키텍처 패턴이다.
MSA 개념을 프론트엔드 영역에 적용한 것으로 하나의 거대한 SPA 대신 여러 개의 작은 애플리케이션들이 유기적으로 조합되어 전체 서비스를 구성하는 방식이다.

이 말을 왜 했냐면, 
기존 모놀리식 프론트엔드 아키텍처는 애플리케이션의 규모가 커지며 문제가 생기기 시작했다.
- **배포 의존성** - 작은 변경사항에도 전체 애플리케이션을 다시 빌드하고 배포
- **기술 스택 제약** - 모든 팀이 동일한 프레임워크와 버전을 사용
- **코드 충돌** - 여러 팀이 동시에 작업할 때 머지 충돌 빈번하게 발생
- **느린 빌드 시간** - 코드 베이스 커질수록 빌드와 테스트 시간 증가

그렇다면 코드 분리로도 해결할 수 있지 않은가?
명확하게 디렉토리 구조를 나누고 모듈을 분리한다고 해도 단일 빌드 프로세스, 단일 빌드 파이프라인 문제를 해결하기 쉽지 않음

이 중 모노레포는 **소스 코드 관리 전략**에 해당한다.

```text
Monorepo  → 하나의 Repository에 여러 프로젝트
Multirepo → 프로젝트마다 개별 Repository
```

그렇다면 어떤 방식을 선택해야 할까? 선택하기 전 고려해야 할 사항들이 존재한다.

## 언제 빌드 타임에 통합할 것인가?
어떤 시점에 빌드 파일들을 통합할 것인지?
- **npm을 통한 라이브러리화 | 모노레포 기반 통합(Nx, Turborepo)**
  - 소규모 팀 | 단일 팀에서 모든 마이크로 앱을 관리하는 경우
  - 초기 로딩 성능이 중요하고 간단한 배포 및 설정을 선호하는 경우(Toss 보면 이것도 마냥 간단하지 않은 듯)

## 언제 런타임에 조립할 것인가?
- **런타임을 통한 조립 Module Federation | Next.js의 Multi-zone 경로 기반 통합**
  - 여러 팀이 독립적으로 개발하고 배포해야 하는 경우
  - 각 마이크로 앱의 배포 주기가 다른 경우
  - 대규모 조직에서 팀 간 의존성을 최소화하려는 경우

## 어디에서 조립할 것인가? (Server / Client / Edge)
- SSR 처럼 서버 측 구성(Podium | tailor 등)
  - (장점) 초기 로딩이 빠르고 SEO 최적화
  - (단점) 모든 조립 작업이 서버에서 일어나다 보니 서버 부하가 커지고 캐싱 전략도 복잡해짐
- CSR Module Federation 
  - 런타임 시점에서 클라이언트가 조립할 수 있게
  - (장점) 필요한 부분만 선택적으로 로드 해 인터렉티브한 UX 제공, 서버 부하 줄일 수 있음
  - (단점) 초기 로딩이 느리고 SEO 최적화도 별도 작업 필요
- 에지 측 구성(Edge Computing) 여러 개의 독립적인 마이크로 앱을 에지(Edge) 네트워크에서 조합하여 사용자에게 최종 페이지를 전달
  - Cloudflare Workers나 Lambda@Edge 같은 플랫폼이 지원
  - (장점) 서버 부하 분산, 빠른 응답 속도 제공
  - (단점) 에지 컴퓨팅 인프라 구축, 디버깅과 모니터링 복잡, 상대적으로 높은 인프라 비용 발생



정답은 없다.
회사의 조직 구조, 팀 규모, 배포 전략, 그리고 서비스 특성에 따라
Monorepo와 Multirepo, 그리고 마이크로프론트엔드의 적용 방식은 달라질 수밖에 없다.

중요한 것은 유행이 아니라, 
우리 팀이 감당할 수 있는 복잡도와 실제로 해결하려는 문제다.
