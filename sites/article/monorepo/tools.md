## 모노레포 초기 세팅

모노레포를 구성한다고 해서  
어떤 도구 하나만 골라서 쓰면 끝나는 건 아니다.

실제로는  
**역할이 다른 여러 도구들을 조합해서 하나의 구조를 만든다**고 보는 게 맞다.

이 챕터에서는  
“그래서 결국 뭘 써야 하는데?”라는 질문에 답하기 위해,  
모노레포를 구성할 때 자주 등장하는 도구들을  
**역할 기준으로 정리해봤다.**


## 1. Workspace  
### (의존성 관리 레이어)

Workspace는  
**패키지를 어떻게 설치하고,  
프로젝트 간 의존성을 어떻게 연결할지**를 책임지는 레이어다.

모노레포의 가장 기본이 되는 구성 요소라고 볼 수 있다.

### Yarn Workspaces

- 가장 기본적인 모노레포 구성 방식
- 모든 프로젝트가 하나의 의존성 그래프를 공유
- 설정이 단순하고 진입 장벽이 낮음

👉 소규모 프로젝트나 개인 프로젝트에 잘 어울린다.

---

### pnpm Workspaces

- `node_modules`를 직접 복제하지 않고 symlink 기반으로 관리
- 디스크 사용량이 크게 줄어듦
- 의존성 접근 규칙이 엄격해서 실수를 빨리 발견할 수 있음

👉 중·대규모 프로젝트나  
의존성 관리에 신경 써야 하는 팀에 적합하다.


## 2. 모노레포 오케스트레이터  
### (빌드 / 실행 관리 레이어)

Workspace 위에서  
**빌드, 테스트, 린트, 캐시, CI/CD 흐름**을 관리해주는 도구들이다.

프로젝트 수가 늘어날수록  
이 레이어의 존재감은 점점 커진다.

### Nx

- 단순한 도구라기보다는 모노레포를 위한 “플랫폼”에 가까움
- 변경된 프로젝트만 빌드 / 테스트
- 강력한 로컬 · 원격 캐시
- 프로젝트 간 의존성 그래프를 기반으로 한 실행 관리

**장점**
- 대규모 조직에서 매우 강력
- CI/CD 최적화에 특히 유리

**단점**
- 러닝 커브가 높은 편
- 설정과 규칙이 많아 초반에는 다소 무겁게 느껴질 수 있음

👉 플랫폼 팀이 있거나,  
대규모 모노레포를 전제로 한다면 좋은 선택지

- 참고: [올리브영 | Nx를 활용한 마이크로 프론트엔드 구축](https://oliveyoung.tech/2025-07-22/what-is-MFE-part1/)

---

### Turborepo

- Vercel에서 만든 모노레포 빌드 오케스트레이터
- Next.js 전용은 아니며 프론트엔드 생태계 전반에서 사용 가능
- Task 기반 파이프라인
- 증분 빌드, 병렬 실행, 로컬 / 원격 캐시 지원

**장점**
- 설정이 직관적이고 가벼움
- 프론트엔드 중심 모노레포에 잘 맞음

**단점**
- Nx에 비해 제공하는 기능 범위는 제한적

👉 스타트업이나  
FE 중심 팀에서 많이 선택하는 구성

## 3. 레거시 도구

### Lerna

- 과거에는 모노레포의 사실상 표준
- 현재는 `Workspace + Nx / Turborepo` 조합에 밀린 상태
- 신규 프로젝트에서는 권장되지 않음

👉 기존 레거시 모노레포 유지보수 용도로만 의미가 있다.

## 정리

모노레포를 구성할 때  
도구들을 이렇게 나눠서 생각하면 정리가 쉽다.

- **Workspace**  
  → 의존성 설치 방식과 패키지 연결 담당  

- **Nx / Turborepo**  
  → 빌드, 테스트, 캐시, CI/CD 실행 흐름 담당  

결국 중요한 건  
“어떤 도구가 더 좋다”가 아니라,

👉 **우리 프로젝트 규모와 팀 구조에 어떤 조합이 맞는가**다.
